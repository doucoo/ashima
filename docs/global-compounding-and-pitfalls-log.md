# 全局复利与踩坑日志

## 目的

这个文档用于在 `ashima` 项目体系内长期记录：

- 犯过的错误
- 踩过的坑
- 走过的弯路
- 未来可以持续复用的方法

目标不是写流水账，而是让未来做类似任务时，能更快、更稳、更少返工。

## 记录原则

### 1. 记录事实，不写空话

每条记录至少说明：

- 当时做了什么
- 错在哪里
- 为什么会错
- 下次如何避免

### 2. 优先记录高复用价值问题

优先记录这些内容：

- 会重复出现的配置坑
- 容易误判的行为模式
- 容易被忽略的前置检查
- 已验证有效的稳定方法

### 3. 一条记录要能独立复用

每条日志最好包含：

- 场景
- 症状
- 根因
- 修复方式
- 未来 SOP 化建议

## 推荐记录模板

```text
### [编号] 标题
- 场景：
- 问题现象：
- 根因：
- 修复方法：
- 以后怎么避免：
- 是否可沉淀为 SOP：是 / 否
```

## 日志区

### [001] 在错误项目范围里落规则文档
- 场景：本应先在 `ashima` 项目中试行规则，却把文档和入口规则先写到了 `hermes-agent` 仓库
- 问题现象：污染了不该改动的项目范围
- 根因：执行前没有再次确认“当前规则先在哪个项目试运行”的边界
- 修复方法：撤销 `hermes-agent` 中的误改，并将全部规则文档改写到 `ashima` 仓库内
- 以后怎么避免：任何涉及“全局规则”的写入，都先确认作用域是“当前仓库内全局”还是“跨项目全局”
- 是否可沉淀为 SOP：是

### [002] 把“内部流程存在”误当成“用户看得见过程”
- 场景：多智能体/多视角流程设计中，虽然内部做了 critique 或 review，但用户只能看到最后摘要
- 问题现象：用户感知不到真正的辩论过程，输出像并列观点而不是交锋
- 根因：没有把中间回合设计成用户可见产物
- 修复方法：把中间回合显式结构化输出，例如 initial / critique / unresolved disagreements / synthesis
- 以后怎么避免：设计多智能体协作时，区分“内部步骤”和“对用户可见的过程展示”
- 是否可沉淀为 SOP：是

### [003] 项目材料散落在多个位置
- 场景：调研、脚本、截图、临时文档分散在不同目录
- 问题现象：交接困难、复盘困难、后续查找成本高
- 根因：没有强制项目目录隔离规则
- 修复方法：规定每个项目必须创建独立目录，所有相关材料全部放入对应项目文件夹
- 以后怎么避免：新项目启动时先建目录，再开始产出任何材料
- 是否可沉淀为 SOP：是

## 复利方法区

### 方法 1：先定义目录，再开始执行
适用场景：所有新项目

做法：

1. 先确定项目编号和名称
2. 创建独立项目目录
3. 先创建基础文档结构
4. 再开始调研、实现、产出内容

长期价值：

- 防止资料散落
- 降低后期整理成本
- 提高复盘和复用效率

### 方法 2：先做对比研究，再做结构改造
适用场景：技能设计、工作流设计、多 agent 交互设计

做法：

1. 先找已有方案
2. 拆出对方真正有效的结构特点
3. 区分“能力存在”和“用户感知到的能力”
4. 再做本地改造设计

长期价值：

- 避免盲改
- 避免只学表面形式
- 提高改进命中率

### 方法 3：把可复用错误写成 SOP 或规则
适用场景：重复出现的坑、返工、误判

做法：

1. 发现问题
2. 记录根因
3. 抽象成规则
4. 写入全局文档或 SOP

长期价值：

- 错误不会白犯
- 系统会越来越稳
- 后续项目启动成本更低
